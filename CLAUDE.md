# CLAUDE.md

This file provides guidance when working with code in this repository.

## Build Commands

`Common.xcodeproj` is generated by XcodeGen from `project.yml` and is not committed. Run `make generate` once after cloning, and again whenever `project.yml` changes.

```bash
# Generate Xcode project (required before opening in Xcode or running xcodebuild)
make generate
# or directly:
xcodegen generate

# Build via Xcode (framework target)
xcodebuild build -scheme Common -project Common.xcodeproj -destination 'generic/platform=iOS Simulator' CODE_SIGNING_ALLOWED=NO

# Build DemoApp
xcodebuild build -scheme DemoApp -project Common.xcodeproj -destination 'generic/platform=iOS Simulator' CODE_SIGNING_ALLOWED=NO

# Build XCFramework (device + simulator, output to XCFramework/)
make build_xcframework
# or directly:
./build_xcframework.sh
```

The framework targets **iOS 16.0+**, Swift 5.9+. The DemoApp targets iOS 26.0+. There are no unit tests or linting tools configured.

## Distribution

The project is distributable via **Swift Package Manager**. The `Package.swift` defines a single library product (`Common`) with one target. It is also buildable as an Xcode framework (`Common.framework`) via the generated `.xcodeproj` (see XcodeGen above).

## Architecture

This is both an **Xcode framework project** and an **SPM package**. The `Common` target produces `Common.framework`; `DemoApp` is a sample app showing usage across 8 modules (Alerts, DeclarativeUI, Extensions, Home, LocalAuth, Networking, Onboarding, Storage).

### MVVM-C Pattern

- **Coordinators** (`Architecture/`): `BaseCoordinator` manages navigation flow and child coordinator hierarchies.
- **ViewControllers** (`ViewControllers/`): `BaseViewModelableViewController` provides standard lifecycle and automatic ViewModel binding.
- **Views** (`Views/`): `BaseView` and `BaseCell` use `@UIViewBuilder` for declarative UIKit layouts.
- **Protocols** (`Protocols/`): 35+ protocols (`Navigationable`, `AlertPresentable`, `ViewModelable`, `Actionable`, `DismissRequestable`, etc.) define the contracts between layers.

### Declarative UI DSL

Views are built using `@UIViewBuilder` (a result builder) with `VStack`, `HStack`, `VList`, `HList` container types defined in `Views/`. This enables SwiftUI-like layout syntax in UIKit — use `.snap(to:)` to constrain to a superview.

**Important**: When overriding `mainView` in a view controller, prefer `lazy var` properties for complex view hierarchies. Using computed `@UIViewBuilder` properties with `.with {}` closures and `setConstraints` can cause layout failures because a new view instance is created on each access, detaching subviews from the hierarchy.

### Networking (`Network/`)

`BaseClient` provides the HTTP networking base. `HTTPService` handles actual requests. Implement endpoints conforming to the networking protocols and call `request(from:_:result:)`. Supports multipart uploads.

### Storage (`Storage/`)

Three typed storage options: `KeychainWrapper` (secure), `KeyValueStore` (UserDefaults-backed), `FileStorage` (file system).

### Extensions (`Extensions/`)

400+ extensions across 49 categories (UIKit, Foundation, AVFoundation, CoreGraphics, Vision, etc.). When looking for a utility, check here before adding new code — the extension likely already exists.

### Utils (`Utils/`)

System integration managers: `AppleLoginManager`, `LocalAuthenticationManager` (FaceID/TouchID), `CameraManager`, `NotificationRegisterManager`. Also: `Logger`, `Debouncer`.

### Global Types

- `Global.swift`: Dispatch helpers, `EmptyResult` type alias
- `TypeAliases.swift`: 83+ type aliases for result handlers and protocol compositions

## DemoApp Module Patterns (Reference: bioidentity-v3)

### Module Structure

Each DemoApp module follows a consistent MVVM-C structure:

```
Module/
├── ModuleViewController.swift   # UI, inherits BaseViewModelableViewController<ViewModelProtocol>
├── ModuleViewModel.swift        # Logic, conforms to ViewModel + ViewLifecycleable
├── ModuleWireframe.swift        # Factory enum, creates and wires module
└── Cell/ (optional)             # BaseViewModelableCell<CellViewModelProtocol> subclasses
```

### Wireframe Factory Pattern

Every module has a Wireframe enum with a static `createModule` factory:

```swift
enum FooWireframe {
    static func createModule(with delegate: BaseModuleDelegate, onAction: @escaping Handler<ActionType>) -> UIViewController {
        let viewModel = FooViewModel(delegate: delegate, onAction: onAction)
        return FooViewController(viewModel: viewModel)
            .with { viewModel.view = $0 }
    }
}
```

Key points:
- Wireframe is always an `enum` (no instances)
- `delegate` is typically `BaseModuleDelegate` (or a composition like `BaseModuleDelegate & RetryRequestable`)
- `.with { viewModel.view = $0 }` wires the bidirectional VC <-> VM reference
- Result handlers use `Handler<T>` type alias

### ViewModel Conventions

```swift
protocol FooViewModelProtocol: ViewModel, ViewLifecycleable {
    // module-specific requirements
}

final class FooViewModel {
    private weak var delegate: BaseModuleDelegate?
    private let onAction: Handler<ActionType>
    weak var view: FooViewProtocol?

    init(delegate: BaseModuleDelegate, onAction: @escaping Handler<ActionType>) {
        self.delegate = delegate
        self.onAction = onAction
    }
}

extension FooViewModel: FooViewModelProtocol {}

extension FooViewModel: ViewLifecycleable {
    func onViewDidLoad() {
        view?.addBackButton { self.delegate?.onGoBackRequested() }
        view?.setupNavigationBar()
    }
    func onViewWillAppear() { /* start sessions, reload data */ }
    func onViewWillDisappear() { /* stop sessions, cleanup */ }
}
```

Key points:
- `weak var delegate` communicates up to coordinator
- `weak var view: ViewProtocol?` communicates down to ViewController
- `onAction`/`onResult` closures communicate results back to coordinator
- ViewLifecycleable methods are called automatically by BaseViewModelableViewController

### View Protocol Pattern

Define what the ViewModel can call on the ViewController:

```swift
protocol FooViewProtocol: BackButtonAddable, NavigationBarSetupable {
    func set(title: String)
    func reloadList()
}
```

Common protocol adoptions: `BackButtonAddable`, `NavigationBarSetupable`, `CameraSessionHandler`, `Vibrator`, `ActivityIndicatorable`, `KeyboardDismissable`, `ScreenSizeMeasurable`

### Coordinator Navigation

```swift
final class AppCoordinator: BaseCoordinator {
    override func start() {
        let vc = FooWireframe.createModule(with: self) { action in
            switch action {
            case .next: self.push(nextVC)
            case .child: self.addChildAndStart(ChildCoordinator(navigationController: self.navigationController))
            }
        }
        set(vc)  // set as root
    }
}
```

Navigation methods from `Navigationable`:
- `set(vc)` / `set([vcs])` — replace root
- `push(vc)` — push onto stack
- `pop()` / `pop(.toRoot)` / `pop(.to(vc))` — pop navigation
- `present(.overCurrent, viewController: vc)` — modal
- `dismiss()` / `dismiss { completion }` — dismiss modal

### Declarative UI Building Reference

**mainView with @UIViewBuilder:**
```swift
@UIViewBuilder override var mainView: UIView {
    VStack(alignment: .center, margins: .init(top: 16, left: 16, bottom: 16, right: 16), spacing: 24) {
        titleLabel
        HStack(distribution: .fillEqually, spacing: 8) { button1; button2 }
        list
    }
}
```

**Lazy var for subviews (required for stateful views):**
```swift
private lazy var titleLabel = UILabel("Title")
    .font(.systemFont(ofSize: 24, weight: .bold))
    .textAlignment(.center)
    .textColor(.black)

private lazy var list = VList()
    .register(MyCell.self)
    .dataSource(self)
    .delegate(self)
    .setConstraints { $0.snap(to: $1) }
```

**Constraint methods:**
- `.snap(to:)` — fill parent (all edges)
- `.snapLeadTopTrail(to:)` / `.snapLeadBottomTrail(to:)` — three edges
- `.snapLeadTrail(to:)` — horizontal edges only
- `.alignCenter(with:)` — center in parent
- `.setWidth(to:, multiplier:)` / `.setHeight(to:, multiplier:)` — relative sizing
- `.set(width:)` / `.set(height:)` — absolute sizing
- `.setRatio()` / `.setRatio(widthToHeight)` — aspect ratio (default 1:1)
- `.pinTop(to:)` / `.pinBottom(to:)` — pin to specific anchor

**Fluent styling:**
- `.backgroundColor(_:)`, `.textColor(_:)`, `.font(_:)`, `.textAlignment(_:)`
- `.round(_:)` / `.setAsRoundedView(radius:)`, `.shadow(color:radius:offset:opacity:)`
- `.isHidden(_:)`, `.contentMode(_:)`, `.numberOfLines(_:)`
- `.onTap { }` — tap gesture handler
- `.with { }` — generic configuration closure

**VStack/HStack parameters:**
- `alignment:` — `.fill`, `.center`, `.leading`, `.trailing` (maps to UIStackView)
- `distribution:` — `.fill`, `.fillEqually`, `.fillProportionally`, `.equalSpacing`
- `margins:` — `UIEdgeInsets` for padding
- `spacing:` — CGFloat between arranged subviews

**Common UI components:**
- `VList` / `HList` — UICollectionView wrappers (vertical/horizontal)
- `ActionButton` — styled primary action button
- `CardView` — card container with viewModel
- `AlertView` — modal alert with viewModel
- `Footer` — reusable footer with image
- `CircularActivityIndicatorView` — animated loading spinner
- `Snackbar.show(_:)` / `Toast.present(_:)` — notification feedback
- `PreviewView` — camera preview container

### Networking Module Pattern

```swift
// 1. Environment
enum PostEnvironment: Environment {
    case posts
    var baseURLAsString: String { "https://api.example.com" }
}

// 2. Endpoint
enum PostEndpoint: Endpoint {
    case getPosts
    var baseURL: URL { PostEnvironment.posts.baseURL }
    var path: String { "/posts" }
    var method: HTTPMethod { .get }
    var headers: HTTPHeaders? { nil }
    var parameters: Encodable? { nil }
}

// 3. Client
final class PostClient: BaseClient {}
extension PostClient {
    func fetchPosts(result: @escaping NetworkResultHandler<[Post]>) {
        request(PostEndpoint.getPosts, result: result)
    }
}
```

### Storage Module Pattern

```swift
// Model must conform to Storable (which is Codable)
struct MyItem: Storable { let value: String; let timestamp: Date }

// Three store types:
let userDefaultsStore = KeyValueStore(type: .notSecure(.userDefaults))
let fileStore = KeyValueStore(type: .notSecure(.files))
let keychainStore = KeyValueStore(type: .secure)

// Typed storage with enum keys:
struct MyStorage: SingleRawValueKeyValueObjectStorage {
    var type: KeyValueStore.StoreType { .secure }
    enum Keys: String { case myItem }
}
extension MyStorage {
    func add(item: MyItem) { add(item: (.myItem, item)) }
    func get() -> MyItem? { get(using: .myItem) }
    func delete() { remove(using: .myItem) }
}
```

### Custom Alert (Modal) Pattern

```swift

let vc = CustomAlertWireframe.createModule(
    VStack(alignment: .center, margins: .init(top: 16, left: 16, bottom: 16, right: 16), spacing: 24) {

```

### UseCase Protocol Pattern (from bioidentity-v3)

Business logic is encapsulated in UseCase protocols with default implementations:

```swift
protocol FetchDataUseCase: AnyObject {
    func fetchData(using id: String, result: @escaping NetworkResultHandler<DataResponse>)
}

extension FetchDataUseCase {
    func fetchData(using id: String, result: @escaping NetworkResultHandler<DataResponse>) {
        MyClient().fetchData(id: id, result: result)
    }
}
```

Coordinators conform to UseCase protocols, getting default implementations for free. This keeps coordinators clean while centralizing business logic.

## Production Usage Notes (from bioidentity-v3 & UniPay)

### Layout Preference

Production strongly favors **VStack/HStack** with `@UIViewBuilder` over manual constraints. `.setConstraints` is used only at the top-level container. Prefer stack-based composition for new UI code.

### Most Used Extensions

The fluent styling chain is the core API surface in production: `.font()`, `.textColor()`, `.backgroundColor()`, `.cornerRadius()`, `.numberOfLines()`, `.onTap {}`. These appear across 60+ files.

### Button Pattern (Production Standard)

Production uses `UIButton(configuration:)` with `.filled()`, `.borderless()`, `.plain()` — not `ActionButton` directly:

```swift
UIButton(configuration: .filled()
    .attributedTitle(.init("Title", attributes: .init()
        .with { $0.font = .appFont(style: .bold, size: 14) }))
    .baseBackgroundColor(.black)
    .baseForegroundColor(.white)
    .cornerStyle(.capsule)
).onTap { ... }.setRatio(327/60)
```

### TextField Pattern

```swift
UITextField()
    .borderColor(.gray600).borderWidth(1)
    .font(.appFont(size: 16)).textColor(.black)
    .placeholder("...", color: .gray600, font: .appFont(size: 16))
    .setAsRoundedView(radius: 4).setRatio(327/56)
    .onEditingChanged { ... }
```

### Card Container Pattern

Use VStack + `.round()` + `.shadow()` — not CardView:

```swift
VStack(margins: ..., spacing: 16) { content }
    .backgroundColor().round(radius: 16)
    .shadow(color: .black.withAlphaComponent(0.15), offset: .init(width: 0, height: 4), opacity: 1, radius: 15)
```

### Constraint Patterns (Most Common)

```swift
// Full safe area
.setConstraints { $0.snap(to: $1.safeAreaLayoutGuide) }
// Safe area top + full bottom
.setConstraints { $0.snapLeadTopTrail(to: $1.safeAreaLayoutGuide); $0.snapBottom(to: $1) }
// Keyboard-aware
.setConstraints { $0.snapLeadTopTrail(to: $1.safeAreaLayoutGuide); $0.pinBottom(to: $1.keyboardLayoutGuide.topAnchor) }
```

### Aspect Ratio Convention

Use `.setRatio(designWidth/designHeight)` from design specs (e.g., `.setRatio(327/60)` for buttons).

### Dynamic View Updates

Use `didSet` on properties to trigger view rebuilds:

```swift
private var selectedItem: Item? {
    didSet { container.removeArrangedSubviews(); container.views { updatedItems } }
}
```

### Known Component Gaps

The following are commonly needed but not yet in Common — consumers create their own:
- **PaddingLabel** — `UILabel` with configurable `UIEdgeInsets` padding
- **GIFImageView** — `UIImageView` subclass supporting animated GIF display

### Underused Features

These exist in Common but see little to no production use:
- `.with {}` closure pattern (use direct property chaining instead)
- `ActionButton` (use `UIButton(configuration:)` instead)
- `CardView` (use VStack + `.round()` + `.shadow()` instead)
- `MultipartRequest` / `ParameterEncoder`
- `NotificationRegisterManager`, `LocationAuthorizationManager` (domain-specific)

## Known Gotchas

- **`onMoveToSuperview` lifecycle**: The `setConstraints` extension relies on swizzled `didMoveToSuperview`. If a view already has a superview when `setConstraints` is called, the handler executes immediately. Be aware of this when debugging layout issues.
- **`UIViewBuilder` and `loadView`**: `BaseViewController.loadView()` sets `self.view = mainView`. Since `mainView` is a computed property, avoid expensive or stateful setup in it — use `lazy` container views instead.
